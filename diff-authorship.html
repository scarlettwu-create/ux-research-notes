Diff Views for Human Edits vs AI Edits — Competitive Research

Context. For the AIRO SDK connector editor flow (PMO-2980). Follow-up to the highlight-definition note (airo-ai-vs-human-edit-highlighting) which flagged authorship attribution as unresearched. The question: when human edits and AI edits mix in the same file, does any product distinguish them in the diff view, and should AIRO? This doc covers both layers: the pending review diff (before accept) and persistent attribution (after accept/commit).

Products covered. Cursor, GitHub Copilot (VS Code), Zed, aider, Claude Code, Git AI.

Scope. How each product's diff surface treats mixed human + AI edits, whether authorship (who wrote this line) is recorded and shown, at what granularity (line, commit, round), and what users say about it.

Product selection. Same finding as the undo/redo research: there is no meaningful Tier 1, because workflow automation products (Zapier, n8n, Make) have no in-editor AI diff surface at all. The mixed-authorship problem only exists once a product embeds an AI-editing code surface, which is what AIRO is adding.

Product Tier Why it's relevant
GitHub Copilot (VS Code) 2 The incumbent review flow; its pending-diff overlay is the best-documented mixed-edits failure
Cursor 3 The only product that shipped line-level AI-vs-human attribution as a product feature (Cursor Blame) plus an open spec (Agent Trace)
Zed 3 Cleanest hunk-level pending review; a control case with no authorship layer
aider 3 The commit-level attribution pioneer: "(aider)" author suffix on by default since early on
Claude Code 3 The Co-Authored-By trailer that made AI attribution a mainstream (and contested) practice
Git AI 3 Open-source line-level attribution infrastructure (git notes), shows what write-time recording looks like

Part 1 — Behavior per product

Two distinct layers keep coming up, so each product is described on both: (a) pending diff — the review surface before accept. (b) persistent attribution — any record of authorship after accept/commit.

Cursor

(a) Pending diffs make no authorship distinction, and the review pane actively mishandles the mix: "Review: All Changes shows the original changes that were made by the agent, disregarding the manual changes that the user did" (Cursor forum, November 2025). (b) Cursor Blame (shipped in 2.4, Enterprise plan only, admin-enabled) extends git blame with three authorship classes per line (Tab, Agent with model name, Human), hover shows the conversation that produced the line, and the commit view shows percentage attribution per source. Cursor also published Agent Trace, a draft RFC for a vendor-neutral JSON trace format classifying code ranges as human, AI, mixed, or unknown, with content hashes to survive refactoring; it deliberately leaves merges, rebases, and large agent changes unresolved.

GitHub Copilot (VS Code)

(a) The pending-edits overlay is computed from the chat session's last checkpoint, so a manual edit made outside the session is claimed as an AI edit and can't be cleared: "Even if I manually edited those files outside the chat session... I cannot clear the diff overlay" (vscode #307439, April 2026). The requested fix in that thread is a "accept the current workspace state as the new baseline" command. Staging auto-accepts all pending AI edits, mixing them permanently. (b) No authorship layer; once kept, AI edits are indistinguishable from typed ones.

Zed

(a) The Agent Panel offers the cleanest hunk-level accept/reject in the set, but hunks carry no authorship metadata; a user edit inside a pending hunk simply becomes part of the hunk. (b) None. Zed is the control case: good review mechanics, zero attribution.

aider

(b, its whole model) aider marks authorship at commit time: by default it appends "(aider)" to the git author name when it wrote the changes, and to the committer name when it merely committed the user's dirty files, a two-way distinction (wrote it vs touched it) no other product makes. --attribute-co-authored-by switches to a Co-authored-by trailer, and --no-attribute-author turns it all off. There is no pending diff layer; git is the review surface.

Claude Code

(b) Adds a Co-Authored-By trailer to commits and a footer to PR descriptions by default. The practitioner critique is that the trailer is the wrong tool: "The Co-Authored-By trailer in git was meant for people who exchanged drafts, and that's not what's happening here" (Fabio Rehm, March 2026), proposing dedicated Coding-Agent and Model trailers instead because the same model runs behind many tools. (a) In the VS Code extension, proposed diffs are gated per edit, but accepted edits carry no mark.

Git AI

(b) Agent Blame is open-source infrastructure for exactly this: agents call git ai checkpoint while coding, checkpoints become an Authorship Log attached to the commit as a git note, recording which model, agent, and prompt generated each line, and attributions are rewritten through rebases, squashes, and merges so they survive history editing. It works because attribution is recorded at write time by the tool that made the edit, not inferred afterward.

Part 2 — Themed insights

Theme 1: In the pending diff, nobody separates your edits from the AI's, and interleaving breaks the review

Every product with a pending-review surface treats the diff as one undifferentiated set of changes. When the user edits during review, the products fail in two directions: Copilot claims the user's edits as AI edits (vscode #307439), and Cursor shows the AI's original changes while "disregarding the manual changes that the user did" (Cursor forum). Zed avoids the bugs but only by making no claim at all.

For AIRO: this confirms the highlight-note recommendation. A pending highlight that claims to mean "AIRO wrote this" will be wrong in exactly the interleaved case the connector workflow guarantees (prompt, review, refine by hand, test). Defining the highlight as "changed since checkpoint" is the only definition the diff can actually keep.

Theme 2: Persistent attribution exists, but it lives at the commit boundary, not in the diff

The products that do record authorship all record it when code crosses a durable boundary: aider stamps the git author field, Claude Code stamps the commit message, Git AI attaches a git note at commit, and Cursor Blame reads committed code. Nobody attributes authorship inside the live buffer or the pending diff. The one thing users ask Cursor for is pushing attribution further downstream (into the PR view), not upstream into the editor.

For AIRO: the analogous durable boundary is the checkpoint/version, not the line. A version history entry that says "this round was AIRO, from this prompt" is the industry-consistent place to record authorship, and AIRO already has that metadata for free.

Theme 3: Attribution gets doubted the moment it ships, and after-the-fact inference can't back it up

Cursor Blame's launch thread is a catalog of trust failures: "it still shows no AI attribution" (Mstoner), "Every file it just says 'No blame data available for this file'" (Joe_Thompson), no conversation summary or thread link (jduffin_adobe), and an accuracy challenge from the same user: why did a PR show "30%" AI credit "when all changes were made by the Cursor Agent" (Cursor forum, January-February 2026). The academic result explains why this is hard to fix after the fact: on realistic mixed files, the best line-level detector reaches an F1 of only 0.56 (HybridCodeAuthorship), so authorship that wasn't recorded at write time is effectively unrecoverable. Git AI's whole design (agent self-reports at write time, log survives rebases) is the working answer.

For AIRO: if authorship attribution is ever shipped, it has to come from AIRO's own edit pipeline recording what it wrote, at the moment it writes. That's cheap for AIRO (every AI edit already flows through one pipeline) and impossible to retrofit later.

Theme 4: Attribution is socially contested: some users demand it, others strip it

A study of 613 GitHub snippets and 111 practitioners found 76.6% of developers consistently or occasionally self-declare AI-generated code, citing maintenance tracking and ethical responsibility, while 23.4% never do, arguing heavy modification makes original authorship meaningless (Self-Declaration of AI-Generated Code). The product record mirrors the split: aider ships attribution on by default but documents the off switches, an entire genre of guides exists for disabling Claude Code's trailer, and Cursor Blame's thread includes users who want to opt out individually after team-wide enablement (Bernard_Ye) or turn it off entirely (Ying_Jianghong). Attribution touches contribution graphs, review politics, and blame in the human sense of the word.

For AIRO: any attribution surface should be a governance/team setting, not a hardcoded label. Cursor gating Blame to Enterprise with admin enablement is the deliberate positioning: attribution as an audit feature for organizations, not a badge on every line for every user.

Part 3 — AIRO design implications

Competitor pain / pattern What it could mean for AIRO
Pending diffs claim user edits as AI edits or drop them (Copilot, Cursor) Worth keeping the pending highlight defined as "changed since checkpoint," never "AIRO wrote this"; the second claim breaks on interleaved edits
The requested Copilot fix is a re-baseline command Worth making every accept/save re-baseline the diff automatically, so stale-overlay state can't exist
All shipped attribution sits at a durable boundary (commit/author/note), none in the buffer Worth putting authorship on the checkpoint/version entry ("Round 3, by AIRO, from this prompt") rather than on lines
After-the-fact detection maxes out at F1 0.56 on mixed files If line-level attribution is ever wanted, record it from AIRO's edit pipeline at write time; don't plan on inferring it later
Attribution accuracy gets challenged immediately (Cursor Blame launch) Worth showing the evidence with the claim: link the version entry to the prompt and diff that produced it, the way Cursor links blame to conversations
Attribution is contested; users disable trailers and opt out Worth making any authorship display a workspace/admin setting, consistent with Workato's governance posture
aider distinguishes "AI wrote it" from "AI committed it" If AIRO ever saves/releases on the user's behalf, worth keeping those two facts separate in the audit trail
Cursor ships Blame as Enterprise-only governance Signal that AI authorship audit is an enterprise buying concern; relevant to AIRO's agent-governance story beyond the editor

The biggest gap surfaced by this research

No product distinguishes human from AI edits inside the pending diff, and the two that try to reconcile mixed edits there (Copilot, Cursor) generate the loudest complaints in the space. Everyone who succeeded at attribution moved it to a durable boundary and recorded it at write time. For PMO-2980 this resolves the open question from the highlight note in a specific way: the diff view should stay authorship-blind ("changed since checkpoint"), and authorship should live one level up, on the checkpoint/version entry, where AIRO already knows the answer with certainty. The open question for the PRD is whether that checkpoint-level attribution is surfaced at all in v1 (a "by AIRO" marker on version history entries costs almost nothing and matches where the industry landed), and whether Workato's governance surface eventually wants the audit-grade version (per-round prompt + diff linkage), which is what Cursor sells as Enterprise Blame.

Part 4 — Research references

  • On Developers' Self-Declaration of AI-Generated Code (Xu et al.): 76.6% of developers self-declare AI code at least occasionally; motivations are maintenance tracking and ethics; non-declarers cite heavy modification and overhead.
  • HybridCodeAuthorship: benchmark of realistic mixed human/AI Python files; best detector reaches F1 0.48 chunk-level / 0.56 line-level, showing after-the-fact authorship recovery is unreliable.
  • Agent Trace RFC (Cursor, February 2026): draft open spec for recording AI contributions (human / AI / mixed / unknown per code range, storage-agnostic); the emerging interchange format if AIRO ever exports attribution.

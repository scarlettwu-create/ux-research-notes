Distinguishing Human Edits from AI Edits — Two Directions Compared

Context. For the AIRO SDK connector editor flow (PMO-2980). The team has two related decisions in play: the in-editor highlight model (current decision: highlight only AIRO's edits, user edits stay plain) and the deep view (current decision: show all changes in a round, no author distinction). This doc lays out both directions side by side: who ships each, what users complain about, what each buys you, and where the technical bodies are buried. Sources are drawn from the diff-lifecycle, undo/redo, and diff-authorship research in this collection.

Products covered. Cursor, GitHub Copilot (VS Code), Zed, aider, Claude Code, Git AI, plus Google Docs and git/PR review as the mature-convention baselines.

Scope. The comparison is about edit surfaces: live editor highlights, pending review diffs, state-vs-state diffs (deep view, version history), and persistent attribution (blame/commit level).

Direction 1 — Distinguish human edits from AI edits

Who does it

Every AI-native editor already does this in the live review surface, by design. Pending highlights in Cursor, Copilot, Zed, Windsurf, and JetBrains mark the AI's proposed edits only; the user's own typing is never highlighted. Accept/Reject controls apply to the AI's edits, and the user's edits ride the native undo stack. The AIRO editor model the team decided (highlight = AIRO's edits, user edits plain, Accept/Reject for AIRO + native Undo for the user) matches this industry default exactly.

A smaller set persists the distinction after acceptance:

  • Cursor Blame (2.4, Enterprise-only, admin-enabled): git blame extended with per-line Tab / Agent / Human classes, model names, and links to the conversation that produced each line; commit view shows percentage attribution.
  • aider: appends "(aider)" to the git author name by default, and uniquely separates "AI wrote it" (author) from "AI committed it" (committer).
  • Claude Code: Co-Authored-By trailer on commits and a PR footer.
  • Git AI Agent Blame (open source): the agent self-reports checkpoints while coding; attribution becomes a git note recording model, agent, and prompt per line, rewritten through rebases and squashes.
  • Agent Trace (Cursor RFC, February 2026): draft open spec classifying code ranges as human / AI / mixed / unknown.

The mature human-collaboration precedent also distinguishes: Google Docs colors changes per author in version history. Multi-author attribution is a convention users already understand; AI authorship extends it.

What users complain about

The complaints cluster on the distinction breaking, not on the distinction existing:

  • Copilot claims the user's manual edits as AI edits and can't clear them: "Even if I manually edited those files outside the chat session... I cannot clear the diff overlay" (vscode #307439, April 2026). The requested fix is a re-baseline command.
  • Cursor shows the AI's stale changes while "disregarding the manual changes that the user did" (Cursor forum, November 2025).
  • When attribution shipped as a feature, accuracy was doubted immediately: "Every file it just says 'No blame data available'" and a challenge to a 30% AI-credit figure "when all changes were made by the Cursor Agent" (Cursor Blame launch thread, January-February 2026).
  • Attribution is socially contested: users ask to opt out individually after team-wide enablement (same thread), a genre of guides exists for stripping Claude Code's trailer, and practitioners argue the trailer is the wrong mechanism entirely ("meant for people who exchanged drafts", Fabio Rehm).

Benefits

  • Review focus. The user reviews only what the AI did; their own edits don't need their own approval. This is the entire logic of the accept/reject gate.
  • Clean control semantics. Accept/Reject manages AI edits, native Undo/Redo manages user edits, and the two never collide. This is the team's decided model, and it's why "Undo" was renamed "Reject all."
  • Audit and governance. "Which lines did the AI write" is an enterprise buying concern; Cursor sells Blame as Enterprise-only. For Workato's governance story this is familiar territory.
  • Norm alignment. 76.6% of developers self-declare AI-generated code at least occasionally, citing maintenance tracking and ethics (arXiv 2504.16485).

Technical problems

  • The interleaved-edit ambiguity is unavoidable. The moment a user types inside a pending AI region, "whose line is this" has no clean answer. Every product hits this; it's the team's open 1C item. The two workable mechanisms both resolve ambiguity by collapsing it: typing anywhere auto-accepts the round (Chisin's direction; Cursor's commit behavior is the analog), or touching a highlighted line auto-accepts that line (Scarlett's direction). Both keep the surviving highlights honestly pure-AI.
  • Event-log tracking drifts. Tracking "the AI edited these ranges" as remembered state is the root cause of the phantom-highlight complaint class (highlights surviving accepts, restarts, renames; see diff-lifecycle Theme 1). Any distinction layer needs to be reconciled against the actual file, not just recorded.
  • Attribution can't be recovered later. On realistic mixed files the best line-level detector reaches F1 0.56 (HybridCodeAuthorship). If it isn't recorded at write time from the AI's own pipeline, it's gone.
  • History editing breaks naive attribution. Rebases, squashes, and refactors detach line ranges from records; Git AI rewrites its notes through these, and Agent Trace explicitly leaves them unresolved.

Direction 2 — Don't distinguish: one diff, all changes

Who does it

  • Git and PR review, the deepest convention in the space. A PR diff is state-vs-state (branch vs base) and author-blind; reviewers review the change, not the author. Workato's own Recipe Diff and Zapier's draft-vs-published compare are the same model.
  • Every AI editor at the durable boundary. Copilot: staging auto-accepts all pending edits; Cursor: committing auto-accepts diffs. Once code crosses into version control, all four products stop distinguishing.
  • The AIRO deep view as decided: documents all changes in one round against the checkpoint, user and AIRO edits alike, all highlighted.
  • Reject all as decided: reverts the whole state to the checkpoint, including the user's own edits, precisely because the live state can't be split by author reliably.

What users complain about

The complaints cluster on losing track and losing work:

  • Losing track of what the AI did. In the CHI 2026 comparison study, agent-mode participants reduced effort but lost awareness of changes: "I did not expect 10 files... 1000 lines each" (Code with Me or for Me?). Once edits mix, the reviewer can't tell what needs scrutiny.
  • Review burden lands on whoever reads the mixed diff. "AI agents frequently make unnecessary changes that reviewers ultimately reject, adding noise to the review process and eroding trust" (RDEL #129); the "Endless Stream of AI Slop" analysis calls this externalizing the producer's speed gain onto the reviewer (arXiv 2603.27249).
  • All-or-nothing revert eats the user's own work. The known complaint shape from the checkpoints research: "there are situations where we might have partially successful code updates, but then extra unwanted elements get introduced in the later stages" (Cursor forum). AIRO's decided Reject-all behavior (user's line disappears with the round) is exactly this shape; the mitigation is that native Undo recovers it, which the spec should promise explicitly.
  • The demand signal for attribution exists because mixing destroys it. Cursor Blame, Git AI, agentblame, and the Agent Trace spec all exist because teams asked "which of this was AI" after the fact and got no answer.

Benefits

  • Robust by construction. A state-vs-state diff (current vs checkpoint) is computed, not remembered. The entire phantom-highlight complaint class cannot occur. This was the strongest design implication in the diff-lifecycle research.
  • No accuracy disputes. A diff that claims nothing about authorship can't be wrong about authorship. Cursor Blame's launch-week credibility fight is not a risk this direction takes.
  • Matches the deep view's job. The deep view documents what changed in a round so it can be reviewed and restored; for that job, authorship is irrelevant and completeness is everything. One missing user edit in the deep view would be worse than no attribution at all.
  • Cheap and predictable. Two-state diff is a solved problem; no per-keystroke bookkeeping, no reconciliation logic, no attribution storage.

Technical problems

  • Authorship is irreversibly lost at mixing time. If any future need (governance audit, "what did AIRO do last sprint") wants authorship, it must have been recorded at write time; after-the-fact inference is a research-grade unsolved problem (F1 0.56, above).
  • Reject-all must be conservative. Because the state can't be split by author, rejecting the AI's work necessarily takes the user's interleaved work with it. That's defensible only with a reliable recovery path (one-step native Undo) and honest copy.
  • The review gate needs another mechanism to scope itself. If highlights don't say "this is AIRO's," something else must define what Accept/Reject applies to. The checkpoint boundary does this job: everything since the checkpoint is the reviewable unit.

How the two directions actually combine

The industry pattern, and the pair of decisions the team has already made, is not either/or. It's distinguish in the live review surface, don't distinguish in state-vs-state surfaces:

Surface Direction Why
Live editor highlights Distinguish (AIRO edits only) Review focus + clean Accept/Reject vs native Undo split; matches every AI editor
Deep view (round diff vs checkpoint) Don't distinguish Completeness is the job; computed diff is robust; matches git/PR convention
Reject all Don't distinguish (whole state reverts) Live state can't be split by author; recovery = native Undo
Version history / checkpoint entries Distinguish at round level ("this round was AIRO, from this prompt") Where the industry successfully records authorship: durable boundary, write-time, certain

The load-bearing piece is the open 1C mechanism: what happens when the user edits a highlighted line before accepting. Both candidate answers (typing auto-accepts the round; touching a line auto-accepts that line) work by keeping the highlight layer honestly pure-AI, which is what separates AIRO's model from the Copilot/Cursor complaint generators, whose highlights silently absorb or discard user edits. Whichever 1C option wins, the spec sentence that matters is: a highlight is always a real, current AIRO-authored difference, or it is gone.
